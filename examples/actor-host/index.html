actor-host-demo.wasm (kototama.contract's actor:host ABI, partial)

Imports clock_monotonic/log_write/sha256_hex (module "kotoba", same convention kotoba.wasm-exec and kototama.tender use). Wired via actor-host.js -- a browser-side port of kotoba-lang/kototama's kototama.contract (HostCaps/RuntimeLimits/ validate-import-surface, pre-flight AND per-call grant checks, same as the JVM tender). This demo element runs on the main/DOM thread (like KotobaWasmElement always has), so it only wires the 7 imports that can run there. http-post now has a real implementation too -- createHttpPostBridge, a SharedArrayBuffer+Atomics.wait blocking bridge to a companion Worker that does the actual fetch -- but it requires the guest itself to be instantiated inside a Worker (browsers disallow Atomics.wait on the main thread) plus a cross-origin-isolated page (COOP/COEP headers), neither of which this main-thread demo element provides. See src/actor-host.js's header comment and test/verify-http-post-bridge.mjs for the bridge itself verified end-to-end; wiring a Worker-hosted KotobaWasmElement variant that can actually use it from a page is a follow-up, not done here.

crypto-demo.wasm (gen_keypair/sign/verify -- vendored ed25519)

Imports gen_keypair/sign/verify (same fixture shape kototama.tender's (JVM) tender_test.clj compiles via wasm-tools). The guest generates a keypair, signs a 4-byte message, and verifies its own signature -- main() returns 1 iff verification succeeds. Ed25519 math is real arithmetic over bytes already in memory, not I/O, so it can be genuinely synchronous without touching the always-async Web Crypto SubtleCrypto API -- see src/vendor/README.md for where the actual (vendored, unmodified) @noble/curves comes from.
